## 0. 철학

COAT는 세 가지 원칙을 항상 준수한다.

| 원칙 | 설명 |
|------|------|
| **사용자 친절성** | AI가 강제하지 않는다. 사용자가 항상 최종 결정권을 가진다 |
| **일치 검증** | 설계와 구현이 일치할 때까지 반복한다. 타협은 기록으로 남긴다 |
| **가상 팀** | 에이전트가 각자의 역할로 실제 팀처럼 협업한다 |

## 2. 6단계 사이클

```
PLAN → DESIGN → CAST → LOOP → ALIGN → WRAP
                          ↑_____________↓ (ALIGN 실패 시 반복)
```

| 단계 | 역할 | 산출물 |
|------|------|--------|
| PLAN | 목표 정의, 요구사항 작성 | PLAN 문서 |
| DESIGN | 구조 설계, 컴포넌트 정의 | DESIGN 문서 |
| CAST | 팀 구성 (에이전트 배정) | 팀 구성표 |
| LOOP | 반복 개발 + 검증 | 구현 결과, 스냅샷 |
| ALIGN | 기획자 + 검증자 합의 판단 | 합의 결과 |
| WRAP | 완료 처리, 백로그 정리, 회고 | 백로그, PR |

## 3. 팀 구성 (CAST)

### 3-1. 기본 팀 (프로젝트 단위 고정)

| 에이전트 | 역할 | 주요 산출물 |
|---------|------|------------|
| 기획자 | 요구사항 정의, 방향 결정, 팀 조율 | PLAN/DESIGN 문서, 이슈 우선순위 |
| 개발자 | 실제 구현, 버그 수정 | 코드, 수정 결과 |
| 검증자 | 사용자 시나리오 시뮬레이션, 이슈 발견 | 검증 리포트, 이슈 목록 |

```
기본 3명은 프로젝트 내내 동일한 에이전트 유지
  → 프로젝트 맥락, 코드베이스, 품질 기준 일관성 보장
  → 기능마다 CAST를 새로 하되 기본 3명은 교체 없음
```

### 3-2. 동적 전문가 (기능 단위, CAST마다 새로 배정, 최대 2명)

| 전문가 | 투입 키워드 |
|--------|-----------|
| UX 전문가 | modal, animation, onboarding, form, wizard, drag, gesture, 화면전환, 인터랙션 |
| 데이터 모델러 | relation, foreign key, join, migration, schema, 정규화, 다대다, 계층구조 |
| 통합 전문가 | webhook, OAuth, REST API, SDK, third-party, payment, 소셜로그인, 외부연동 |

**투입 판단 알고리즘 (A+C 방식):**

```
Step 1. 체크리스트 (기획자가 DESIGN 읽고 답변)
  Q1. UI 인터랙션이 3개 이상인가?     Y → UX 전문가 후보
  Q2. 외부 API/서드파티 연동이 있는가? Y → 통합 전문가 후보
  Q3. DB 관계가 3개 이상인가?          Y → 데이터 모델러 후보

Step 2. 예시 참조 (엣지케이스 보완)
  소셜 로그인          → 통합 전문가
  대시보드 차트        → UX 전문가
  게시판               → 기본 3명으로 충분
  결제 + 복잡한 UI     → 통합 + UX (2명 동시)

Step 3. 기획자 최종 확정
  2명 초과 시 핵심 복잡도 높은 전문가 우선 선택
```

**LOOP 중 전문가 교체 없음** — 팀은 CAST 시점에 확정, 끝까지 유지

### 3-3. 수동 전문가 (사용자 명시 요청 시만)

| 전문가 | 투입 조건 |
|--------|----------|
| 보안 전문가 | 기능 완성 후 사용자가 보안 검토 요청 시 |

### 3-4. 팀 사이즈 규칙

```
최소: 3명 (기본 팀)
최대: 5명 (기본 3 + 동적 전문가 최대 2)
보안 전문가: 5명 제한 외 별도 (수동 요청 시)
```

## 4. LOOP 알고리즘

### 4-1. 종료 조건 (모두 충족 시 ALIGN 진입)

```
1. 수치 기준
   기능 97% + UX 85% + 속도 80% 모두 충족

2. 안정성 기준
   연속 3라운드 동안 새로운 이슈 없음

3. 최소 라운드
   12라운드 이상 완료
```

### 4-2. 라운드별 검증 깊이

```
Round  1~5:  기본 검증 (기능 동작 확인)
Round  6~10: 엣지 케이스 검증
Round 11+:   심층 품질 모드
```

### 4-3. 스냅샷 스케줄

3라운드마다 기획자/검증자 번갈아 작성. 사용자 개입 없음.

```
Round 3  → 기획자 스냅샷
Round 6  → 검증자 스냅샷 + 사용자 체크포인트
Round 9  → 기획자 스냅샷
Round 12 → 검증자 스냅샷 + 사용자 체크포인트
Round 12+ → 3라운드마다 반복
```

**스냅샷 포맷:**

```
[기획자 스냅샷 - Round N]
- 완료된 요구사항
- 미완료 요구사항
- 방향 수정 필요 여부

[검증자 스냅샷 - Round N]
- 통과 항목
- 잔여 이슈 목록
- 현재 Match Rate (기능/UX/속도)
```

### 4-4. 사용자 체크포인트 (Round 6, 12, 18...)

```
"현재 상태입니다. 계속할까요?"
  [계속] [방향 수정] [중단]

[방향 수정] 선택 시:
  사용자 입력 → 기획자 반영 → LOOP 재개
```

### 4-5. 검증 리포트 포맷

```
[검증 리포트 - Round N]
기능:  ✅/⚠️/❌  (97% 기준)
UX:   ✅/⚠️/❌  (85% 기준)
속도:  ✅/⚠️/❌  (80% 기준, 감지만)
```

## 5. ALIGN 알고리즘

기획자 + 검증자 동시 합의로 LOOP 종료 여부 판단.

```
ALIGN_FAIL 1회 → LOOP 재개 (정상)
ALIGN_FAIL 2회 → 기획자가 요구사항 재검토
ALIGN_FAIL 3회 → 사용자 개입 강제

사용자 선택지:
  [요구사항 수정]          → 수정 후 LOOP 재개
  [조건부 통과 → P1* 이월] → 미해결 이슈 P1*로 이동 (빚 기록)
  [중단]                   → WRAP으로 강제 종료
```

## 6. 미니 COAT 사이클 (P1* 처리)

P1*는 ALIGN 3회 실패 후 조건부 통과된 이슈. 반드시 별도 사이클로 처리.

### 6-1. 구조

```
일반 COAT:  PLAN → DESIGN → CAST → LOOP(12+) → ALIGN → WRAP
미니 COAT:                  CAST → LOOP(6+)  → ALIGN → WRAP

- P1* 하나당 미니 사이클 하나 (격리 처리)
- 팀: 기본 3명만 (전문가 추가 없음)
- LOOP 최소: 6라운드
- 조건부 통과 불가 (반드시 ALIGN 통과 필요)
```

### 6-2. 미니 COAT ALIGN 실패 시

```
ALIGN_FAIL 1회 → LOOP 재개
ALIGN_FAIL 2회 → 기획자 문제 재정의
ALIGN_FAIL 3회 → 사용자 개입 강제

사용자 선택지:
  [요구사항 축소] → 범위 줄이고 LOOP 재개
  [포기]          → 백로그에서 제거 + 기록 보존
  (조건부 통과 없음)
```

### 6-3. 결과 기록 (성공/실패 모두)

```
[P1* 결과 기록 - {이슈명}]
─────────────────────────
상태: ✅ 해결 / ❌ 실패

--- 시도 기록 ---
시도 1 (Round 1~6):
  방법: {시도한 방법}
  결과: 기능 N% / UX N% / 속도 N%
  실패 이유: {구체적 원인}

시도 2 (Round 7~12):
  방법: {시도한 방법}
  결과: 기능 N% / UX N% / 속도 N%
  결과: ✅ ALIGN 통과 / ❌ 실패 이유

--- 해결 요약 (✅ 시) ---
핵심 원인: {근본 원인}
해결 방법: {결정적으로 효과있던 방법}
적용 가능한 유사 케이스: {비슷한 상황 설명}
─────────────────────────
```

저장 위치:
```
.coat/audit/p1-records/
  ├── {이슈명}-{날짜}.md   ← 성공/실패 모두 기록
  └── _INDEX.md            ← 전체 목록 + 해결률
```

## 7. WRAP 알고리즘

### 7-1. 백로그 분류

| 등급 | 내용 | 출처 |
|------|------|------|
| P1* | 조건부 통과 이월 이슈 | ALIGN 실패 3회 |
| P1 | 이번 LOOP 미해결 잔여 이슈 | 검증자 리포트 |
| P2 | 검증자가 "아쉬움" 표시한 개선사항 | 검증자 리포트 |
| P3 | 기획자가 "나중에" 표시한 확장 아이디어 | 기획자 판단 |

### 7-2. 다음 PLAN 연결

```
P1* → ⚠️⚠️ 강력 경고 + 미니 COAT 사이클 (조건부 통과 불가)
P1  → ⚠️ 경고: "이전 미해결 이슈" [반영] [무시] (사용자 결정)
P2  → 제안으로 표시
P3  → 선택 목록으로 표시
```

## 8. GitHub 연동 (선택, 기본 비활성)

### 8-1. 제1 규칙: 가독성 우선

```
모든 Issue/PR/Commit:
  1. 핵심 내용 최상단
  2. 세부 내용 하단
  3. 불필요한 내용 제거
```

### 8-2. 단계별 자동화

```
PLAN  → Issue 자동 생성 (auto_issue = true 시)
          라벨: coat:plan
          포맷: ## 핵심 목표 / ## 주요 변경사항 / ## 상세

CAST  → Branch 자동 생성 (auto_branch = true 시)
          이름: coat/{기능명}/{날짜}

LOOP  → Commit 사용자 요청 시만
          메시지: [COAT-{Round}] {핵심 변경 한 줄}
                  - {세부1}
                  - {세부2}

WRAP  → PR 자동 생성 (auto_pr = true 시)
          포맷: ## 완료된 것 / ## 변경 내역 / ## 백로그
          라벨: coat:wrap
        → 백로그 Issue 자동 등록 (auto_issue = true 시)
          P1*: coat:p1-star
          P2:  coat:p2
          P3:  coat:p3
          (P1은 다음 PLAN에서 사용자 결정 — Issue 생성 안 함)
```

**라벨 자동 생성:** 라벨이 없으면 `gh label create`로 자동 생성 후 진행.

### 8-3. 설정

```json
// .coat/config.json
{
  "github": {
    "enabled": false,
    "repo": "",
    "auto_issue": true,
    "auto_branch": true,
    "auto_pr": true
  }
}
```

## 9. 프로젝트 상태 파일 구조

```
{프로젝트 루트}/
└── .coat/
    ├── config.json           ← GitHub 연동 등 설정
    ├── state/
    │   ├── memory.json       ← 현재 단계, 팀, Match Rate, 체크리스트
    │   ├── backlog.json      ← P1*/P1/P2/P3 목록
    │   └── history.json      ← 완료된 기능 목록 (WRAP마다 추가)
    ├── snapshots/
    │   ├── round-3.md        ← 기획자 스냅샷
    │   ├── round-6.md        ← 검증자 스냅샷
    │   └── ...
    └── audit/
        └── p1-records/
            ├── {이슈명}-{날짜}.md
            └── _INDEX.md
```

**`history.json` 스키마:**

```json
{
  "items": [
    {
      "feature": "기능명",
      "completedAt": "YYYY-MM-DD",
      "matchRate": { "기능": 97, "UX": 85, "속도": 80 },
      "backlog": { "p1star": 0, "p1": 1, "p2": 2, "p3": 1 }
    }
  ]
}
```

## 10. Web Dashboard (v2.0.0)

COAT 진행 현황을 실시간으로 시각화하는 로컬 웹 대시보드.

### 10-1. 실행

```bash
# 저장소 루트에서
npm run dashboard -- --project /path/to/your/project

# --project 생략 시 현재 디렉토리 기준
npm run dashboard
```

> http://localhost:3030 에서 접근

### 10-2. 구성 요소

| 패널 | 표시 내용 |
|------|----------|
| Phase Bar | 6단계 진행 상태 (완료/현재/미래) |
| LOOP 진행 | 현재 라운드 + 기능/UX/속도 게이지 |
| 체크리스트 | 완료/미완료 항목 |
| 백로그 | P1*/P1/P2/P3 목록 |
| 스냅샷 | 라운드별 스냅샷 (클릭 시 전문 보기) |
| 완료 히스토리 | WRAP 완료된 기능 목록 + Match Rate |

### 10-3. 기술 구성

```
dashboard/
├── server.js       ← Node.js HTTP 서버 (npm 의존성 없음)
└── dashboard.html  ← 단일 HTML 파일 (폴링 방식, 2초 간격)
```

- `.coat/` 디렉토리를 직접 읽어 API 제공 (`/api/state`, `/api/snapshot/{name}`)
- 외부 의존성 없음 — Node.js 기본 모듈만 사용
